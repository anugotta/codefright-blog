First used coroutines in 2019 on a project. Had callback hell - fetching user data, then posts, then comments, all nested three levels deep. Code was unreadable, debugging was difficult.

Decided to try coroutines. Took a while to understand them. Once I got it, couldn’t go back. Been using them in production for years across multiple apps. Sharing what I learned.

What Are Coroutines?#

Coroutines are lightweight threads that can pause and resume. They can suspend execution without blocking the thread. When you call a network API or read from a database, the coroutine pauses, lets other work happen, then resumes when data is ready.

During that pause, the coroutine isn’t consuming CPU. Just waiting - for network response, file I/O, or delay to finish. Meanwhile your app can do other things. This keeps your UI responsive on Android.

Getting Started#

Add the coroutines dependency. Using version 1.10.1 here, check for latest:

dependencies {
    implementation "org.jetbrains.kotlinx:kotlinx-coroutines-android:1.10.1"
}

Your First Coroutine#

Simple example to get started:

class MainActivity : AppCompatActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        
        // Launch a coroutine in the lifecycle scope
        lifecycleScope.launch {
            // This is suspended code
            delay(1000) // Wait for 1 second
            Log.d("Coroutines", "Hello from the future!")
        }
    }
}

Dispatchers#

Dispatchers tell coroutines which thread pool to use. Main ones you’ll use:

suspend fun fetchUserData() {
    // For network calls, file I/O - this is what you'll use most
    val userData = withContext(Dispatchers.IO) {
        apiService.getUser(userId) // Network call happens here
    }
    
    // For CPU-heavy stuff like sorting large lists, parsing complex JSON
    val processedData = withContext(Dispatchers.Default) {
        userData.process() // Heavy computation
    }
    
    // For UI updates - must be on Main thread
    withContext(Dispatchers.Main) {
        binding.textView.text = processedData.name
    }
}

Use Dispatchers.IO for network and database operations. Dispatchers.Default for CPU-intensive work. Dispatchers.Main for UI updates. Main dispatcher is default when launching from lifecycleScope.

Coroutine Builders: launch vs async#

Two main ways to start coroutines:

// Use launch when you don't need a result back - "fire and forget"
lifecycleScope.launch {
    saveUserPreferences() // Just do the work, don't care about return value
}

// Use async when you need the result
val userDeferred = lifecycleScope.async {
    fetchUserFromApi() // Returns User object
}

// Later, get the result
val user = userDeferred.await()

// You can also run multiple async operations in parallel
val userDeferred = async { fetchUser() }
val postsDeferred = async { fetchPosts() }
val commentsDeferred = async { fetchComments() }

// Wait for all of them
val user = userDeferred.await()
val posts = postsDeferred.await()
val comments = commentsDeferred.await()

Use async when you need to fetch multiple things in parallel. Much faster than sequential.

Scopes#

Coroutines live in scopes. When a scope is cancelled, all coroutines in it are cancelled too. This prevents memory leaks.

Most common scopes:

// In an Activity or Fragment
lifecycleScope.launch {
    // This gets cancelled when the activity/fragment is destroyed
    fetchData()
}

// In a ViewModel
class MyViewModel : ViewModel() {
    fun loadData() {
        viewModelScope.launch {
            // This gets cancelled when ViewModel is cleared
            fetchUserData()
        }
    }
}

Don’t use GlobalScope - it doesn’t get cancelled automatically, can lead to memory leaks. Always use lifecycleScope or viewModelScope.

Why Use Coroutines#

Main benefits:

Lightweight - A thread can take 1-2MB of memory. A coroutine? Just a few kilobytes. Can run thousands simultaneously.

Structured concurrency - When you cancel a parent coroutine, all children get cancelled automatically. Makes cleanup easier. Example:

viewModelScope.launch {
    try {
        val userInfo = async { fetchUserInfo() }
        val userPosts = async { fetchUserPosts() }
        
        // Wait for both results
        displayUserProfile(userInfo.await(), userPosts.await())
    } catch (e: Exception) {
        // Handles errors from both operations!
        showError("Oops, something went wrong!")
    }
}

Sequential code that’s async - Write code that looks synchronous, but it’s actually asynchronous:

// The callback nightmare I used to write
fetchUserData { user ->
    fetchUserPosts(user.id) { posts ->
        fetchPostComments(posts[0].id) { comments ->
            // Three levels deep, hard to read, harder to debug
            displayComments(comments)
        }
    }
}

// With coroutines - clean and readable
lifecycleScope.launch {
    val user = fetchUserData()
    val posts = fetchUserPosts(user.id)
    val comments = fetchPostComments(posts[0].id)
    displayComments(comments)
}

Reads like normal code, but each call can be async. No nesting, no callbacks. Clean, linear code.

Common Mistakes#

1. Forgetting to cancel coroutines

Launched a coroutine that kept polling an API, forgot to cancel when user navigated away. Memory leak.

2. Wrong dispatcher for network calls

Did this once, app froze. Always use Dispatchers.IO for network calls:

// This will freeze your UI - don't do this!
lifecycleScope.launch(Dispatchers.Main) {
    val data = api.fetchLargeData() // Blocks the main thread
}

// Do this instead
lifecycleScope.launch {
    val data = withContext(Dispatchers.IO) {
        api.fetchLargeData() // Happens on background thread
    }
    // We're back on Main thread here, safe to update UI
    showData(data)
}

3. Using GlobalScope

GlobalScope doesn’t get cancelled automatically, can cause memory leaks. Always prefer lifecycleScope or viewModelScope:

// Bad - might outlive your activity
GlobalScope.launch {
    fetchData()
}

// Good - automatically cancelled when activity is destroyed
lifecycleScope.launch {
    fetchData()
}

4. Touching UI from wrong thread

This will crash your app:

lifecycleScope.launch(Dispatchers.IO) {
    val data = fetchData()
    
    // CRASH! Can't update UI from IO thread
    binding.textView.text = data
    
    // This works - switch back to Main thread
    withContext(Dispatchers.Main) {
        binding.textView.text = data
    }
}

5. Exceptions getting swallowed

Coroutines can silently swallow exceptions. Always wrap risky operations in try-catch:

lifecycleScope.launch {
    try {
        val data = fetchData()
        processData(data)
    } catch (e: Exception) {
        // Log it, show error to user, whatever
        Log.e(TAG, "Failed to fetch data", e)
        showError("Something went wrong")
    }
}

Real Examples#

Loading Images#

Pattern for loading images from URLs:

class ImageLoader {
    fun loadImage(imageUrl: String, imageView: ImageView) {
        // Launch in the Main scope but switch to IO for network
        lifecycleScope.launch {
            try {
                val bitmap = withContext(Dispatchers.IO) {
                    URL(imageUrl).openStream().use { 
                        BitmapFactory.decodeStream(it)
                    }
                }
                // Back on Main thread
                imageView.setImageBitmap(bitmap)
            } catch (e: Exception) {
                // Handle error
                Log.e("ImageLoader", "Failed to load image", e)
            }
        }
    }
}

Cache-First Data Fetching#

Pattern used in most apps. Try cache first, fall back to network:

class DataRepository {
    suspend fun fetchUserData(userId: String): User {
        // Try cache first
        return withContext(Dispatchers.IO) {
            try {
                val cachedUser = database.getUser(userId)
                if (cachedUser.isUpToDate()) {
                    return@withContext cachedUser
                }
                
                // Cache miss or outdated, fetch from network
                val freshUser = api.fetchUser(userId)
                
                // Update cache in the background
                launch {
                    database.saveUser(freshUser)
                }
                
                freshUser
            } catch (e: Exception) {
                // If everything fails, at least try to return cached data
                database.getUser(userId) ?: throw e
            }
        }
    }
}

Wrapping Up#

Coroutines have a learning curve. Once they click, they become second nature. Been using them for years, can’t imagine going back to callbacks or RxJava for simple async operations.

Start simple - use lifecycleScope.launch for basic async work, then explore async, withContext, and advanced patterns. Don’t try to learn everything at once.
